## 当前判断

结合现有实现和最近的联调问题，当前最需要优先解决的不是继续堆功能，而是先把以下三件事做稳：

1. 手机访问链路稳定
2. 校验链路可靠且可诊断
3. 搜索/JS 规则兼容性边界更清晰

如果这三部分不先稳住，后续新增功能会继续放大排查成本。

---

## 优先级总览

| 优先级 | 方向 | 目标 |
|------|------|------|
| P0 | 手机访问与发布稳定性 | 让手机端在同一 Wi-Fi、生产部署、重建发布后都能稳定打开并完成操作 |
| P0 | 校验链路可靠性 | 让深度校验、搜索校验、SSE 进度和失败分析更稳定、更可恢复 |
| P1 | 搜索/JS 规则兼容性 | 扩大可支持书源范围，减少“规则支持不足但原因不明”的情况 |
| P1 | 前端结构与维护性 | 降低 `App.vue` 和重复 SSE 逻辑带来的维护成本 |
| P2 | 批量处理与大文件能力 | 提升高量级使用场景下的性能和可操作性 |
| P2 | 结果展示与导出体验 | 让用户更容易理解结果、重试失败项、在手机上导入 |
| P3 | 个性化与增值能力 | 主题、快捷键、评分、报告等非核心增强 |
| P3 | 生产运维完善度 | 监控、备份、容器化、告警、服务治理 |

## P0 手机访问与发布稳定性

### 1. 首页缓存与静态资源版本稳定性

- 保持 `index.html` 禁止缓存，避免旧 HTML 指向失效的哈希资源
- 静态资源继续使用哈希文件名，HTML 与资源缓存策略分离
- 增加前端启动失败兜底提示，而不是只显示标题或空白页
- 增加资源加载自检信息，便于区分“页面资源失败”和“API 不可达”

说明：
2026-03-30 的手机联调现象表现为“页面能看到网站标题，但应用未正常进入”，这类问题本质上属于发布链路和缓存控制问题，优先级应高于一般 UI 优化。

### 2. 手机真机联调链路标准化

- 统一推荐手机访问 `http://电脑局域网IP:8000`
- 开发文档中继续强调不要使用 `localhost`
- 启动脚本增加更明确的局域网 IP 输出和访问提示
- 启动脚本增加基本自检，例如首页、`/api/health`、静态资源是否可访问
- 增加 Windows 防火墙排查说明和快速检查步骤

### 3. 浏览器兼容性与导入链路确认

- 建立 Android Chrome、系统浏览器、微信内置浏览器的真机兼容记录
- 明确记录哪些浏览器会拦截 `legado://` Scheme
- 为“导出到阅读 App”增加失败兜底说明和排查入口
- 对手机端“临时 JSON 链接”的展示做更明显的备用路径说明

### 4. 发布与回归检查

- 每次前端重建后执行最小回归：首页、上传、URL 解析、深度校验、手机导出
- 为“手机访问正常但操作卡住”建立单独排查清单
- 将手机联调检查项纳入发布前 checklist，而不是只保留在文档里

## P0 校验链路可靠性

### 1. 深度校验与搜索校验的恢复能力（基础能力已落地）

- 已支持失败书源批量重试
- 已支持“仅重试超时/网络错误”的选择
- 已支持暂停/恢复校验能力
- 已支持取消校验后保留已完成结果
- 搜索校验已支持从失败书源数据重试，不必重新上传完整文件

后续增强方向：

- 断线后自动恢复同一个会话，而不是依赖用户重新发起
- 为暂停状态增加更明确的队列长度和已发出请求数量
- 对失败重试结果做更细的前端合并提示

### 2. 智能并发与超时策略（基础能力已落地）

- 已支持 1-64 的灵活并发值，不再只靠固定档位
- 深度校验 SSE 已改为每 16 个首轮样本评估一次，避免单个异常窗口连续降并发
- 只有超时、连接失败等网络错误会触发降并发，HTTP 状态失败不会拖低整批策略
- 均衡模式每次最多降低 25%，并发下限为 8，不再快速下降到 1
- 自定义模式在初始并发允许时最低保持 4，不会因智能降速跌到 3 或 1
- 已支持指数退避和预设模式差异化重试次数
- 网络失败已移至首轮完成后的延后重试队列，退避等待不再占用首轮并发槽
- 均衡模式动态超时上限为 40 秒，健康窗口会逐步恢复到基础 30 秒
- 已提供“快速 / 均衡 / 稳定 / 自定义”预设模式
- 服务日志已记录策略评估指标、最终并发、最终超时、首轮网络失败数和实际重试数
- 已兼容 httpx 英文协议、连接和超时异常名，确保瞬时网络错误进入重试路径

后续增强方向：

- 搜索校验当前已接入预设、指数退避、暂停/恢复和取消保留结果，但仍沿用原有固定任务池；后续可单独重构为与深度校验一致的完全动态调度器
- 增加按域名维度的历史响应时间缓存，避免单个慢站点拖低整批策略

### 3. SSE 稳定性

- 增加 SSE 心跳，降低长连接被中间层断开的概率
- 增加断线自动重连和重连后的状态恢复
- 显示实时速度、队列长度、当前并发数
- 增加会话自动清理和并发任务上限

### 4. 失败分析与可诊断性

- 继续细化失败原因分类，如 DNS、连接拒绝、SSL、超时、重定向异常、规则缺失
- 区分“可修复”与“不可修复”的失败类型
- 为常见失败给出修复建议，而不是只给错误字符串
- 允许按失败分类筛选、导出和重试

### 5. 结果缓存

- 缓存短时间内重复校验的 URL 结果
- 搜索校验增加规则解析缓存
- 对同一批次中的重复目标请求做去重
- 为缓存增加 TTL 和清理策略，避免误用过期结果

## P1 搜索与 JS 规则兼容性

### 1. `java.connect` 能力补齐

- 实现可控范围内的 HTTP 请求模拟
- 支持获取重定向后的最终 URL
- 支持 Cookie 读写与持久化
- 增加更明确的请求构造日志，便于定位规则错误

### 2. 常见加密能力补齐

- 使用 Python 侧能力补齐常见加密函数，如 MD5、SHA256、AES、DES
- 兼容阅读 App 常见的 Java 加密调用方式
- 对不支持的加密路径给出明确提示，而不是模糊失败

### 3. 动态渲染支持

- 评估引入 Playwright 或 Puppeteer 作为可选能力
- 仅对确有需要的书源启用 Headless 模式，避免默认成本过高
- 将“需动态渲染”标识为高级兼容路径，而不是主流程默认逻辑

### 4. 搜索校验结果表达

- 增加搜索结果预览
- 对“搜索”“发现”“搜索+发现”给出更清楚的规则覆盖说明
- 支持为失败项标记“规则存在但执行失败”“规则缺失”“运行环境不支持”

## P1 前端结构与维护性

### 1. 拆分 `App.vue`

- 将单个处理、批量处理、搜索校验、结果区、移动端导出区拆分成更清晰的模块
- 将结果组装逻辑从页面层抽离，避免多处重复拼接 `resultData`
- 将 SSE 连接控制抽成 composable，减少多个模式下的重复代码

### 2. 统一状态与提示机制

- 统一成功、失败、警告、信息提示的来源和生命周期
- 增加 Toast 队列管理，避免连续提示淹没有效信息
- 明确区分“前端错误”“接口错误”“规则错误”“网络错误”

### 3. 移动端文案和交互整理

- 为桌面端和移动端分别优化关键提示文案
- 将“导出到阅读 App”和“打开临时 JSON 链接”做更清楚的主备关系
- 对校验进行中的按钮、结果操作按钮做更一致的禁用和反馈策略

## P2 批量处理与大文件能力

### 1. 批量任务管理

### 2. 大文件处理性能

- 评估流式解析和分块处理，减少大文件的内存峰值

### 3. 前端渲染优化

- 结果列表考虑虚拟列表
- 大批量失败项和重复项展示时减少 DOM 压力
- 按需渲染详细内容，避免一次性展开全部结果

## P3 生产部署与运维

### 1. 安全与访问控制

- HTTPS 与自动续期
- 访问速率限制
- 可选的 IP 白名单或鉴权
- 生产环境反向代理配置

### 2. 服务稳定性

- Gunicorn 多进程部署
- Systemd 或同类服务托管
- 崩溃自动恢复
- 日志轮转与服务日志查看（基础能力已落地）

日志轮转基础能力已经落地：后端会写入 `backend/logs/app.log`，使用标准库 `RotatingFileHandler` 做大小轮转，可在 Linux 服务器和 1Panel 文件管理中直接查看。当前使用 1Panel 运维，小圈子自用场景下，先把日志写清楚、留得住、不会无限增长，比上完整监控系统更实用，也更方便直接定位服务器上“跑慢、失败、反馈未落盘”等问题。

已落地内容：

- 后端输出统一日志文件，例如 `backend/logs/app.log`
- 使用 Python 标准库 `logging.handlers.RotatingFileHandler`，避免增加额外部署依赖
- 保留有限历史日志，例如单文件 5MB 到 10MB，最近 7 到 10 个文件
- 记录每次校验的开始时间、结束时间、总数、有效数、失败数、耗时、并发数、超时时间
- 记录慢任务，例如超过 5 分钟的批次
- 记录反馈提交事件，但不在日志里打印过长反馈正文
- 记录未捕获异常和关键接口错误，便于从 1Panel 直接排查
- 保持日志可直接被 1Panel 文件管理或终端查看

后续增强方向：

- 增加更规范的结构化日志字段，方便以后按 `session_id`、耗时、状态过滤
- 如果访问人数增加，再考虑错误告警或健康检查面板

### 3. 监控与告警

- 健康检查统一化
- 结构化日志
- 错误告警
- 基础性能指标监控

### 4. 交付与回滚

- Docker / Docker Compose
- 版本化部署脚本
- 回滚流程说明
- 配置和导出数据备份策略

## 建议执行顺序

建议按以下顺序推进：

1. 先稳手机访问链路与 HTML/静态资源缓存策略
2. 再补校验链路的重试、缓存、失败分类和 SSE 稳定性
3. 再扩搜索/JS 规则兼容范围
4. 然后拆分前端结构，压低维护成本
5. 最后再补主题、评分、报告、云同步等增值功能

## 结论

项目当前不缺功能点，缺的是一份更聚焦的推进顺序。

近期建议以“手机能稳定打开并跑完整流程”为第一优先级，以“校验链路稳定且可诊断”为第二优先级，以“扩大搜索/JS 规则兼容能力”为第三优先级。只要这三项做扎实，后续的报告、评分、主题、云同步等能力才值得继续投入。
