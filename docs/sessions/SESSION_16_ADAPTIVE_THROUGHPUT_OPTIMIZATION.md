# SESSION 16 - 智能校验吞吐优化

## 1. 会话目标

解决智能均衡模式在部分批次中明显慢于原固定并发的问题，同时保留网络波动下的必要降并发和重试能力。

主要运行环境：

- Linux 服务器
- 2 核 2G 内存
- 约 300 个书源的常用校验批次
- 默认均衡模式：16 并发、30 秒超时

## 2. 根因证据

原控制器从第 8 个样本开始，每完成一个结果都可能继续调整。

使用 32 并发、30 秒超时模拟连续超时结果时，原策略变化为：

```text
32 -> 24 -> 16 -> 8 -> 4 -> 2 -> 1
```

同时存在以下问题：

- HTTP 404 等快速失败也计入总失败率，网络正常时仍会错误降并发。
- 单个任务在内部执行重试和指数退避，睡眠等待期间仍占用并发槽。
- 动态超时可持续增加到 120 秒，失效书源较多时会放大尾部耗时。
- 原日志只记录初始并发和超时，无法确认批次过程中实际降速情况。

## 3. 完成任务

### 3.1 窗口式智能控制

- 最近保留 32 条首轮样本用于观测。
- 每新增 16 条首轮样本形成一个独立决策窗口。
- 每个窗口最多调整一次，不再每完成一个结果连续降并发。
- 只根据超时、连接失败、读写错误等网络错误判断是否降速。
- HTTP 4xx、HTTP 5xx 等快速失败不会触发降并发。

### 3.2 渐进并发与超时边界

- 每次最多降低当前并发的 25%。
- 均衡模式从 16 逐步调整，不会低于 8。
- 健康窗口每次恢复 2 个并发，最高回到初始值。
- 均衡模式动态超时最高为 40 秒。
- 健康窗口每次降低 5 秒，直到恢复基础 30 秒。

### 3.3 首轮优先与延后重试

- 深度校验 SSE 的单个 HTTP 任务只执行一次请求。
- 首轮优先完成全部书源，不允许重试项抢占首轮请求槽。
- 可恢复网络失败进入按时间排序的延后重试队列。
- 指数退避等待保存在队列中，不计入正在运行的 HTTP 任务数。
- 首轮全部结束后才进入重试阶段。
- 均衡模式重试阶段通常使用 8 并发。

### 3.4 进度与日志

- SSE 策略状态增加 `phase`：`primary` 或 `retry`。
- 前端进度区显示“首轮”或“网络重试”。
- 策略评估日志记录错误率、超时率、P75 耗时和调整前后参数。
- 完成日志增加：
  - `final_concurrency`
  - `final_timeout`
  - `primary_network_failures`
  - `retries`

## 4. 技术决策

- 自适应控制器只接收首轮样本，重试结果不会重复影响同一批网络判断。
- 32 条历史窗口仅用于日志和观测，实际决策使用每组新增的 16 条独立样本。
- 重试队列与 HTTP 请求解耦，退避等待不会占用活动并发槽。
- 本轮只优化深度校验 SSE；搜索/发现校验仍保留原固定任务池，后续单独重构。
- 保留现有 API 参数、暂停、恢复、取消和部分结果能力。

## 7. 功能验证清单

- 健康批次下均衡模式是否保持 16 并发
- 快速 HTTP 失败较多时是否仍保持并发
- 网络异常窗口是否只从 16 降到 12，而不是连续降到 1
- 连续异常时均衡模式是否最低保持 8
- 超时时间是否最高显示 40 秒
- 进度区是否显示“首轮”与“网络重试”
- 暂停、恢复和取消是否在两个阶段都生效
- 取消后是否继续保留已完成结果

## 8. 服务器复测方法

部署更新后，使用之前同一批约 300 个书源：

1. 选择均衡模式执行完整校验。
2. 记录总耗时、有效数和失败数。
3. 查看服务日志中的 `智能策略评估` 和 `SSE校验完成`。
4. 重点确认：
   - 健康批次通常保持 16 并发。
   - 若降速，单次只从 16 降到 12。
   - 最终并发不低于 8。
   - 最终超时不高于 40 秒。
   - `primary_network_failures` 与 `retries` 数值合理。
5. 将新耗时与旧智能均衡模式、原固定 16 并发耗时对比。

建议先使用小批次确认功能，再使用完整 300 个书源批次测量真实性能。

## 9. 后续建议

- 根据服务器真实日志微调 35% 网络错误率和 25% 超时率阈值。
- 增加按域名维度的并发限制，避免单个慢域名影响全局判断。
- 将搜索/发现校验重构为相同的首轮优先和延后重试模型。
- 增加 SSE 断线后的会话恢复能力。
