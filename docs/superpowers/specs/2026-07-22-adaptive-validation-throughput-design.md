# 智能校验吞吐优化设计

## 1. 背景

当前深度校验的智能策略在网络波动或失败书源较多时，可能明显慢于原固定并发方案。

已确认的主要原因：

1. 控制器从第 8 个样本开始，每完成一个书源都可能再次调整，导致并发连续下降。
2. 初始 32 并发在连续超时样本下可快速下降为 `32 -> 24 -> 16 -> 8 -> 4 -> 2 -> 1`。
3. HTTP 404、HTTP 500 等快速失败也计入总失败率，网络正常时仍可能错误降并发。
4. 单个书源的指数退避和重试发生在工作任务内部，等待退避期间仍占用调度并发槽。
5. 动态超时可持续增加到 120 秒，配合多次重试会放大尾部耗时。

本次以“均衡模式”为主要优化对象，目标是在 2 核 2G 服务器上保留必要容错，同时让正常批次吞吐接近固定 16 并发。

## 2. 范围

### 2.1 本次包含

- 深度校验 SSE 调度器
- `AdaptiveValidationController` 的并发和超时调整规则
- 深度校验的网络失败延后重试队列
- 智能策略调整日志
- 策略与调度核心单元测试

### 2.2 本次不包含

- 搜索/发现校验动态调度器重构
- 按域名单独限流
- SSE 断线后重新连接和任务恢复
- 跨进程持久化校验会话

## 3. 总体方案

采用“窗口式自适应 + 首轮优先 + 延后重试”。

校验分为两个阶段：

1. 首轮校验：按预设并发优先完成所有书源的第一次请求。
2. 网络重试：只对超时、连接失败、读写错误等可恢复错误进行延后重试。

退避等待由调度器管理，不在正在执行的 HTTP 请求任务中睡眠，因此等待重试不会占用首轮并发槽。

## 4. 自适应并发规则

### 4.1 采样窗口

- 最近样本窗口：32 个首轮请求结果。
- 评估间隔：每新增 16 个首轮结果评估一次。
- 每个评估窗口最多调整一次，禁止每完成一个结果连续降并发。
- 重试阶段不继续训练首轮控制器，避免同一失败书源重复影响网络判断。

### 4.2 错误分类

只使用以下网络错误计算降速指标：

- 超时、连接超时、连接失败
- 读取错误、写入错误、读取超时、写入超时
- 服务器断开、协议错误、网络错误、SSL 错误

HTTP 4xx、HTTP 5xx、规则无效和内容为空不触发降并发，因为这些失败不代表本机并发过高。

### 4.3 降并发

满足以下任一条件时允许降并发：

- 网络错误率不低于 35%。
- 超时率不低于 25%。
- 首轮请求的 P75 耗时不低于当前超时的 80%，且同时存在网络错误。

调整规则：

- 每次最多降低当前并发的 25%。
- 均衡模式从 16 降为 12，再根据后续独立窗口决定是否继续调整。
- 并发下限为初始并发的 50%，且不低于 4。
- 均衡模式最低并发为 8，不允许下降到 1。

### 4.4 恢复并发

一个完整评估窗口同时满足以下条件时逐步恢复：

- 网络错误率不高于 10%。
- 超时率不高于 5%。
- P75 耗时低于基础超时的 60%。

每次增加 2，最高恢复到预设初始并发。均衡模式恢复路径示例：`8 -> 10 -> 12 -> 14 -> 16`。

## 5. 动态超时规则

- 均衡模式基础超时保持 30 秒。
- 只有完整窗口的超时率不低于 35% 时才增加超时。
- 每次增加 5 秒。
- 最大只允许增加到基础超时加 10 秒，均衡模式上限为 40 秒。
- 健康窗口出现后每次降低 5 秒，直到恢复基础超时。
- HTTP 状态失败不会触发超时增加。

## 6. 延后重试队列

### 6.1 首轮请求

- 智能 SSE 调度路径调用单次请求，不在 `validate_source_access` 内部执行重试。
- 首轮始终优先填满当前并发，重试项不得抢占首轮任务槽。
- 首轮成功和不可重试失败立即形成最终结果。
- 可重试网络失败进入延后队列，暂不计为最终失败。

### 6.2 重试阶段

- 所有首轮任务完成后进入重试阶段。
- 重试次数沿用预设：快速 1 次、均衡 2 次、稳定 3 次、自定义 2 次。
- 退避时间沿用指数退避，但等待项不计入正在执行的 HTTP 请求数量。
- 重试并发使用当前并发和初始并发 50% 中的较小值，均衡模式通常为 8。
- 某项重试成功后直接加入有效结果；耗尽重试次数后才加入失败分组。

### 6.3 进度与取消

- `processed` 继续表示已经得到最终结果的书源数量。
- SSE 策略状态增加 `phase`，值为 `primary` 或 `retry`。
- 暂停、恢复、取消对两个阶段都生效。
- 取消时继续保留已形成的最终结果。

## 7. 日志

每次策略调整记录一条结构化服务日志，至少包含：

- session ID
- 当前阶段
- 样本数量
- 网络错误率和超时率
- P75 响应耗时
- 调整前后的并发
- 调整前后的超时
- 调整原因

批次完成日志增加最终并发、最终超时、首轮网络失败数和实际重试数。

## 8. 测试设计

新增 `backend/tests/test_validation_strategy.py`，采用测试先行方式覆盖：

1. 快速 HTTP 失败不会降低并发。
2. 前 15 个网络失败样本不会提前评估。
3. 第一个异常窗口只允许从 16 降到 12。
4. 连续异常窗口不会使均衡模式低于 8。
5. 健康窗口会每次恢复 2 个并发。
6. 动态超时不会超过基础超时加 10 秒。
7. 禁用智能策略时参数完全不调整。
8. 首轮等待时，退避重试项不占用活动请求槽。
9. 首轮全部完成后才开始消费重试队列。

## 9. 验收标准

- 健康批次的均衡模式全程保持 16 并发。
- 快速 HTTP 失败较多时不降低并发。
- 单个异常窗口最多降低一次并发。
- 均衡模式最低保持 8 并发。
- 均衡模式动态超时不超过 40 秒。
- 同一批模拟数据下，智能均衡模式总耗时不超过固定 16 并发基准的 15%。
- 前端生产构建通过，后端策略测试和现有测试通过。
- 部署后使用同一批约 300 个书源对比新旧耗时，并通过服务日志确认实际并发调整过程。

## 10. 兼容性

- 保留现有 API 参数和预设名称。
- 非 SSE 的同步深度校验暂时保留原重试行为。
- 前端无需改变用户操作方式，只在进度区增加当前阶段展示。
- 旧请求未传策略参数时继续使用均衡模式默认值。
